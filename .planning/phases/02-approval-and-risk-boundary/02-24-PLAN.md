<interfaces>
`latch_kill_switch()` currently snapshots recovery scopes only from `order_commands`; with no order it leaves the active set empty. `KillSwitchService.begin_recovery()` already obtains IDs by iterating the ledger-loaded active scopes, so the normal zero-scope call naturally produces `()`. The current SQLite matcher rejects that tuple before checking whether its active scope set is empty.

Retain the existing `RecoveryAssessmentService` and `RecoveryEvidence` contract for every nonempty scope. The new semantics are exact: `assessment_ids == ()` is admissible only when the same immediate transaction reads an empty active scope set and verifies a newly collected `ZeroScopeClearanceProof`. That proof is a separate immutable, ID-free, non-submittable value for the fixed selected Paper Spot target. It canonically captures the current account snapshot and positions, open-order facts, target connection, server-time observation, collection time, and controlled clearance summary. The SQLite verifier must reject unavailable, stale, future-dated, contradictory, or target/account/product-mismatched observations; non-empty open orders, non-zero/residual positions or exposure, pending cancellation work, unresolved local orders, or unresolved submission claims; and any proof that does not canonically round-trip. Begin and complete each collect and bind a fresh proof independently, remain distinct actor-labelled transactions, and append the proof JSON, controlled summary, actor, occurred time, and canonical `[]` to the durable event before the state update commits. An empty tuple with any active scope, duplicate IDs, forged IDs, stale evidence, rejected evidence, or mismatched scope evidence remains rejected; the scoped branch must reject a supplied zero-scope proof and retain its existing assessment validation unchanged.

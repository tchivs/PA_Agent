It currently checks type(scope), type(assessment), scope equality, assessment bindings, accepted status, assessment timestamp, canonical JSON/digest, and ten nonempty top-level mappings, but it does not rehydrate the individual observation types or prove their target/freshness semantics.

Create a domain-only `RecoveryEvidence` validator that owns the exact fixed names `capabilities`, `rules`, `account`, `quote`, `server_time`, `connection`, `open_orders`, `order_rate`, `loss_drawdown`, and `fee_rate`. It must support construction from the service's real typed observations and strict reconstruction from canonical JSON. Reconstruction must reject missing/extra keys, non-object observation payloads, malformed enum/Decimal/datetime values, wrong nested shapes, non-aware timestamps, digest or canonical round-trip mismatches, and observations that do not exactly bind to the persisted `RecoveryScope` target/policy symbol/account/product. It must apply the fixed 60-second rule independently to every observation timestamp, using `window_ends_at` for order-rate evidence.

`RecoveryAssessmentService` remains the normal collector and has no proposal, ticket, command, claim, permit, `OutboundSubmission`, or gateway-submit dependency. The SQLite recorder retains 02-19's same-transaction durable-scope lookup and 02-21's accepted-only ID allocation, but must invoke `RecoveryEvidence.from_canonical_json(...)` before allocation. The two existing begin/complete transactions retain their independent scope-ID, policy, evidence digest, acceptance, and freshness checks. Do not claim that underscore naming is an authorization boundary.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| arbitrary caller -> callable SQLite recovery recorder | A caller can hold an active scope and invoke the underscore recorder with arbitrary RecoveryAssessment JSON. |
| recovery evidence JSON -> durable recovery-assessment row | Serialized data is untrusted until SQLite rebuilds every canonical observation against the transaction-loaded scope. |
| accepted recovery assessment ID -> begin/complete recovery | Even a persisted ID remains untrusted until each immediate transition transaction rechecks active scope coverage, acceptance, bindings, and freshness. |

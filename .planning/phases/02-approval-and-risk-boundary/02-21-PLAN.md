<interfaces>
`SQLiteExecutionLedger.record_recovery_assessment(scope, assessment)` currently accepts a caller-held `RecoveryAssessment`, checks only that the canonical JSON includes the ten observation keys, and mints an ID. `RecoveryAssessmentService.assess(scope)` collects ten gateway calls but returns the same publicly constructible value without controlling persistence. The existing `_recovery_assessment_for()` test helper demonstrates the exploit by passing ten empty objects for an active durable scope.

Replace the public caller-assessment persistence boundary with a narrow controlled persistence path owned by `RecoveryAssessmentService`; do not leave a public `ExecutionLedger` method that accepts a caller-built RecoveryAssessment for ID allocation. The controlled path must validate the exact active scope and preserve the existing two recovery-transition checks. It must also validate each canonical observation's type/target/timestamp and enforce the fixed 60-second window before an accepted recovery record can become durable. A rejected service-produced assessment remains auditable but cannot transition recovery. Do not reuse `ProposalService`, `RiskAssessment`, or `proposal_risk_assessments`.
</interfaces>
</context>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| active RecoveryScope -> recovery assessment service | A scope identity may be visible to callers, but only the service may collect and write a clearance. |
| gateway observations -> durable recovery assessment | Every observation must be canonical, target-bound, complete, and fresh before it becomes an accepted clearance. |
| recovery assessment ID -> durable transition | IDs remain untrusted until each immediate transition transaction revalidates scope, policy, evidence, acceptance, and freshness. |
